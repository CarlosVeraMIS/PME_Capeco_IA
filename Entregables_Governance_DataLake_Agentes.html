A · Governance · 11 entregables · Días 1–20

Gobernanza de la información CAPECO

El governance no es burocracia — es lo que hace que los agentes confíen en los datos que procesan y que CAPECO confíe en lo que MIS hace con su BD.

ℹ️
Punto de partida real: La BD tiene 150 columnas en data_capeco_full. Algunas tienen espacios en el nombre (requieren backtick quoting). MARCA_PINTURA_1 tiene 0% de completitud. MARCA_SANITARIO tiene 100% de registros pero 85% dice "No precisan". Esos son los problemas que el governance debe documentar y resolver antes de que cualquier agente procese esos campos.

Data Contract: CAPECO → MIS (ingesta trimestral)

Días 5–10 DE1PM DoD: firmado por ambas partes antes del Día 12
📄
Cláusulas obligatorias del contrato
  • Schema comprometido: Lista de columnas, tipos y reglas de validación que CAPECO garantiza en cada entrega
  • SLA de entrega: Máximo 15 días calendario después del cierre de cada trimestre (Q1→15 abr, Q2→15 jul, Q3→15 oct, Q4→15 ene)
  • Formato de entrega: Dump MySQL comprimido o acceso directo a Azure MySQL — a definir en el contrato
  • Notificación de cambios de schema: CAPECO notifica con 30 días de anticipación si agrega o elimina columnas
  • Registro histórico: MIS conserva cada entrega trimestral como snapshot inmutable mínimo 7 años
  • Uso permitido: Operación de plataformas PME CAPECO, PME Proveedores y Tasamelo bajo marca MIS×CAPECO

B · Data Lake · 9 entregables · Días 1–20

Arquitectura Azure Data Lake · Bronze / Silver / Gold

El Data Lake no reemplaza el Azure MySQL de CAPECO — se construye encima. El MySQL sigue siendo la fuente de verdad. El lake agrega capas de calidad, versionado y acceso controlado.

🏗️
Arquitectura sobre lo que ya existe: millennialprod.mysql.database.azure.com permanece como fuente primaria. Azure Data Lake Storage Gen2 (ADLS) se provisiona en la misma suscripción Azure de MIS. Los scripts Python actuales (query_pme.py, query_obras.py) seguirán funcionando — solo cambiarán para leer desde la API interna (B7) en lugar de MySQL directo.

Pipeline de ingesta Bronze (Python)
# bronze_ingesta.py — ejecutado por DE2 import mysql.connector, pandas as pd from azure.storage.filedatalake import DataLakeServiceClient import hashlib, json from datetime import datetime # 1. Leer completo desde MySQL query = "SELECT * FROM data_capeco_full WHERE ANIO='2025' AND TRIM='4T'" df = pd.read_sql(query, conn) # 2. Guardar como Parquet particionado path = f"bronze/capeco/raw/year=2025/trimestre=Q4/snapshot.parquet" df.to_parquet(path, index=False) # 3. Generar hash de integridad sha256 = hashlib.sha256(open(path,'rb').read()).hexdigest() manifest = {"rows": len(df), "sha256": sha256, "timestamp": datetime.utcnow().isoformat(), "source": "millennialprod.mysql.database.azure.com/db_capeco"} # 4. Guardar manifest.json junto al parquet
  • Formato: Apache Parquet (columnar, comprimido, compatible con Azure Synapse y Databricks)
  • Inmutabilidad: política de retención de 7 años, sin permisos de delete en el contenedor Bronze
  • Manifest: archivo JSON con COUNT, SHA256, timestamp y nombre de usuario que ejecutó la ingesta
B3
Pipeline Bronze → Silver (limpieza y deduplicación)
Proceso automatizado que transforma el Bronze raw en Silver limpio: deduplicación, validación de tipos, enmascaramiento PII, normalización semántica básica.
Días 10–17 DE1 DoD: Silver tiene 0 duplicados por COD_PROYECTO+TRIM+ANIO · 0 GPS fuera de Lima
⚙️
Transformaciones Bronze → Silver
  • Deduplicación: COD_PROYECTO + TRIM + ANIO → mantiene registro más reciente si hay duplicados
  • Validación GPS: X entre -77.2 y -76.7, Y entre -12.5 y -11.5 (bounding box Lima Metropolitana)
  • Normalización "No precisan": MARCA_SANITARIO, MARCA_PORCELANATO donde valor = "No precisan" / "No informa" / "No tienen" → NULL estructurado
  • Tipos: ANIO como integer, TRIM como string ("Q4"), GPS como float64
  • Enmascaramiento PII-2: TELEFONO PROPIETARIO → hash SHA256 no reversible en Silver
  • Campos con espacios: renombrados en Silver usando snake_case (NOMBRE DEL PROYECTO → nombre_del_proyecto)
Trigger
  • Manual en Sprint 1 (ejecución por DE1)
  • Automático desde Sprint 2: Azure Data Factory o Apache Airflow — ejecuta 24h después de cada Bronze nuevo

B5
Pipeline Silver → Gold (certificación y métricas)

Días 15–20 DE2 DoD: Gold generado · métricas Gold validan vs valores verificados manualmente
⚙️
Métricas calculadas en Gold (no existen en Bronze)
  • PRECIO_X_M2: PRECIO_SOLES / AREA_X_UNIDAD (ya existe en MySQL — se recalcula y valida)
  • ABSORCION_PCT: UNIDADES_A / NRO_UNIDADES × 100 (% unidades ya vendidas)
  • FASE_CODIGO: codificación numérica de FASE para filtros rápidos
  • DISTRITO_NORM: distrito normalizado (sin variantes de mayúsculas/minúsculas)
  • GPS_VALID: flag boolean — coordenadas dentro del bounding box Lima
Separación por producto
  • gold/pme-capeco/ → todos los campos PII-0 + métricas Gold
  • gold/pme-proveedores/ → subset aprobado en A4 (sin PII-1 hasta A10 firmado)

C · Agentes · 8 entregables · Días 1–52

Arquitectura de agentes sobre el Data Lake

Los agentes son procesos Python autónomos que leen del Data Lake, ejecutan una lógica específica y escriben resultados estructurados. No son chatbots. Cada uno tiene un único objetivo, inputs definidos, outputs verificables y manejo de errores explícito.

🏗️
Principio de diseño de los agentes PME: Cada agente es un proceso Python con 4 secciones fijas: (1) validate_inputs(), (2) execute(), (3) validate_outputs(), (4) log_result(). Si cualquier sección falla, el agente termina con error controlado y genera un log JSON — nunca se propaga silenciosamente un dato incorrecto a la capa Gold.

Diseño de arquitectura de agentes
Documento que define la interface común de todos los agentes, el protocolo de comunicación entre ellos y la arquitectura del orquestador.

Interface base de todos los agentes (Python)
from abc import ABC, abstractmethod from dataclasses import dataclass from datetime import datetime import json, logging @dataclass class AgentResult: agent_name: str status: str # "success" | "warning" | "error" timestamp: str inputs_hash: str # SHA256 de los inputs outputs: dict # resultado estructurado errors: list # lista de errores encontrados warnings: list # lista de advertencias metrics: dict # métricas de ejecución class BaseAgent(ABC): def run(self) -> AgentResult: try: self.validate_inputs() # Falla rápido si inputs incorrectos result = self.execute() # Lógica principal del agente self.validate_outputs(result) # Valida que outputs son correctos return self.log_result(result, "success") except Exception as e: return self.log_result({}, "error", error=str(e)) @abstractmethod def validate_inputs(self): ... @abstractmethod def execute(self) -> dict: ... @abstractmethod def validate_outputs(self, result: dict): ...
Principios de diseño
  • Idempotencia: ejecutar el agente dos veces con los mismos inputs produce el mismo resultado
  • Fail-fast: si validate_inputs() falla, el pipeline no continúa — no se propaga data incorrecta
  • Observabilidad: cada ejecución genera un log JSON con timestamp, inputs_hash, métricas y errores
  • Independencia: cada agente puede ejecutarse solo — sin dependencia en memoria compartida
